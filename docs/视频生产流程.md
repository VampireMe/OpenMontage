# OpenMontage 视频生产流程

> 本文档说明：当用户给出一个**任务描述**（主题、参考视频、源素材等）后，OpenMontage 如何一步步产出最终视频。
> 内容基于 `AGENT_GUIDE.md`、`PROJECT_CONTEXT.md`、`pipeline_defs/animated-explainer.yaml` 以及工具注册表的当前配置。

---

## 1. 核心思想：Agent 驱动的指令式管线

OpenMontage 没有“运行时的 Python 编排器”。**AI Agent（Claude Code / Cursor / Copilot 等）本身就是控制平面**，它通过阅读 YAML 管线清单和 Markdown Skill 来驱动生产。

```
用户输入 → Agent 读取 pipeline 清单 → 按阶段读取 stage-director skill
         → 调用 Python 工具 → 生成规范化的 JSON artifact → 自检/人工审批
         → 下一阶段 → 最终 render
```

Python 代码只负责：

- **Tools**：具体能力（生成、分析、合成、处理）
- **Persistence**：checkpoint、artifact、cost log 的读写

所有创意决策、阶段过渡、质量检查、审批策略都写在 **Skill / Manifest** 中。

---

## 2. 输入类型与入口判断

Agent 拿到用户任务后，首先判断它属于哪一类“入口”：

| 输入类型 | 典型描述 | 入口处理 |
|---|---|---|
| **主题/创意型** | “做一个 60 秒解释黑洞的视频” | 直接进入生成类管线（如 `animated-explainer`） |
| **参考视频型** | “像这个 YouTube 视频一样给我做一个” | 先走 `video-reference-analyst`：下载/采样、转录、场景分析、风格拆解，再进入创意提案 |
| **源素材型** | “帮我把这段素材剪成短视频” | 先走 `source_media_review`，再进入 `clip-factory` / `hybrid` / `talking-head` 等素材主导管线 |
| **本地化/配音型** | “把这条视频翻译成中文并配音” | 进入 `localization-dub` 管线 |

---

## 3. 标准生产阶段

大多数生成类管线遵循以下 8 个阶段：

```
research → proposal → script → scene_plan → assets → edit → compose → publish
```

> 部分管线（如 `clip-factory`、`talking-head`）会插入或替换为领域特定阶段（如 `source_media_review`、`clip_extract`）。

### 阶段总览

| 阶段 | 核心目标 | 关键产物（Artifact） | 是否通常需人工审批 |
|---|---|---|---|
| **Preflight（飞行前检查）** | 发现当前机器/环境真实可用工具，呈现能力菜单 | 能力清单、预算基线 | 是（了解可用能力） |
| **research（调研）** | 收集话题背景、数据点、受众疑问、竞品缺口、创意角度 | `research_brief` | 否 |
| **proposal（提案）** | 基于调研给出 3+ 差异化创意方案、制作计划、成本估算、工具路径 | `proposal_packet` + `decision_log` | **是**（关键审批闸口） |
| **script（脚本）** | 把选定方案写成带时间轴的解说词，标注语气、停顿、强调点 | `script` | **是** |
| **scene_plan（分镜规划）** | 把脚本拆解为有序场景，指定每个场景的视觉类型、时长、所需资产 | `scene_plan` | **是** |
| **assets（资产生成）** | 生成/获取配音、图片、视频片段、音乐、图表、代码片段等 | `asset_manifest` | 否（机械阶段） |
| **edit（剪辑决策）** | 将资产组合成时间线：cut、字幕、音乐压幅、转场、锁定渲染引擎 | `edit_decisions` | 否 |
| **compose（合成渲染）** | 按锁定引擎渲染最终视频，混音，输出成片 | `render_report` + `final_review` | 否 |
| **publish（发布）** | 生成平台元数据、章节标记、缩略图概念、导出包 | `publish_log` | **是** |

## 4. 每个阶段的核心事项与工具

以下以 **`animated-explainer`** 管线为例展开。其他管线阶段名和产物可能不同，但逻辑相似。

### 4.1 Preflight — 飞行前检查

在进入任何创意工作之前，Agent 必须先搞清楚：**当前环境到底能用什么工具？**

核心事项：

1. 运行 `registry.provider_menu_summary()` 或 `registry.capability_catalog()`。
2. 检查每个能力族的“已配置/总数”，例如 TTS、图像生成、视频生成、音乐、合成引擎。
3. 检查三大合成运行时是否可用：`ffmpeg`、`remotion`、`hyperframes`。
4. 向用户呈现能力菜单和可解锁项（缺 API key 但配置简单）。
5. 根据可用工具选择/调整管线方案，不承诺无法交付的效果。

**常用工具/查询：**

```python
from tools.tool_registry import registry
registry.discover()
registry.provider_menu_summary()
registry.capability_catalog()
registry.get_by_capability("tts")
registry.get_by_capability("image_generation")
registry.get_by_capability("video_generation")
# 查看 video_compose 支持的渲染引擎
registry._tools["video_compose"].get_info()["render_engines"]
```

### 4.2 research — 调研

核心事项：

- 通过网页搜索、论坛、新闻等收集至少 5 个来源。
- 提炼 3+ 个有依据的创意角度（`angles_discovered`）。
- 找出反直觉数据点、受众误解、市场缺口。
- 所有结论必须 `grounded_in` 具体来源，禁止编造。

产物：`research_brief`（含 `research_summary`、`data_points`、`audience_insights`、`landscape`、`angles_discovered`、`sources`）。

工具：本阶段通常不调用生成工具，以信息收集为主。

### 4.3 proposal — 提案

这是整个流程的**关键审批闸口**：在花钱生成任何资产之前，必须得到用户明确同意。

核心事项：

1. **吸收调研**：从 `research_brief` 提取最有力的角度和数据。
2. **能力校验**：确认 proposal 中提到的工具在 Preflight 中真实可用。
3. **设计 3+ 个真正不同的创意方案**：在 hook、叙事结构、视觉风格、平台、时长上形成差异。
4. **决定视觉风格**：基于主题和受众设计配色、字体、情绪板，必要时生成自定义 playbook。
5. **锁定渲染运行时**：
   - 如果 `remotion` 和 `hyperframes` 都可用，必须同时向用户介绍两者，并等待明确选择。
   - 将 `render_runtime_selection` 记录进 `decision_log`。
6. **制作成本估算**：按工具/操作逐项列出预计花费。
7. **生成参考样片（如适用）**：在参考视频驱动的生产中，先生成 10–15 秒样片让用户确认风格。

产物：`proposal_packet`（含 `concept_options`、`selected_concept`、`production_plan`、`cost_estimate`、`approval`）+ `decision_log`。

工具：本阶段以决策为主，样片环节可能调用 `tts_selector`、`image_selector`、`video_selector`、`video_compose`、`audio_mixer`。

### 4.4 script — 脚本

核心事项：

- 按照选定方案的叙事结构（myth_busting、problem_solution、data_narrative 等）写解说词。
- 控制字数在目标时长 ±10% 以内（通常按 150 WPM 估算）。
- 标注 `enhancement_cues`（音效、画面强调点）。
- 提供 `voice_performance_plan`：语速、停顿、重音、情绪提示。
- 引用调研中的具体数据，避免泛泛而谈。

产物：`script`（含按 section 组织的文本、时长、enhancement cues、voice performance plan）。

工具：通常不调用生成工具，纯文本创作。

### 4.5 scene_plan — 分镜规划

核心事项：

- 把脚本完整覆盖到时间线，无空档、无重叠。
- 每个场景指定类型（`text_card`、`stat_card`、`bar_chart`、`comparison`、`hero_title`、`terminal_scene` 等）。
- 避免连续 3 个同类型场景，保持视觉多样性。
- 明确每个场景所需的资产（图片、视频、图表、代码片段等）。
- 符合已选 playbook 的转场、配色、节奏规则。

产物：`scene_plan`（含场景列表、时长、类型、required_assets、transition）。

工具：通常不调用生成工具。

### 4.6 assets — 资产生成

这是**实际花钱/耗资源**的阶段。所有资产按 `scene_plan` 生成并登记在册。

核心事项：

1. 根据 `production_plan` 选择具体工具（通过 selector 自动路由）。
2. 保持视觉风格一致：使用统一的 style anchors / playbook。
3. 生成配音并核对实际时长；如果超出 scene_plan 允许范围，需退回 script/scene_plan 修改。
4. 生成/获取背景音乐（优先从 `music_library/` 读取，其次调用 `music_gen`）。
5. 记录每个资产的来源、seed、模型版本、成本，确保可复现。
6. 所有文件写入 `projects/<project-name>/assets/`。

产物：`asset_manifest`（含每项资产的路径、生成工具、场景关联、metadata）。

**常用工具：**

| 能力 | Selector / 工具 | 说明 |
|---|---|---|
| 文本转语音 | `tts_selector` | 自动路由到 `google_tts`、`piper_tts`、`elevenlabs_tts` 等 |
| 图像生成 | `image_selector` | 自动路由到 `google_imagen`、`flux_image`、`openai_image`、图库等 |
| 视频生成 | `video_selector` | 自动路由到 `kling_video`、`runway_video`、`veo_video`、本地模型等 |
| 音乐生成 | `music_gen` / `pixabay_music` | 生成或搜索背景音乐 |
| 图表/示意图 | `diagram_gen`（Mermaid）| 流程图、架构图等 |
| 数学动画 | `math_animate`（ManimCE）| 数学/数据可视化动画 |
| 代码片段 | `code_snippet` | 高亮代码展示素材 |
| 分析与处理 | `audio_probe`、`visual_qa`、`audio_energy` | 验证媒体时长、质量、音乐最佳起始点 |

### 4.7 edit — 剪辑决策

核心事项：

- 将资产按时间线切分为 `cuts`，每个 cut 引用 `asset_manifest` 中的有效资产。
- 配置字幕样式（必须与 playbook 兼容）。
- 配置背景音乐：音量、ducking（ narration 出现时压低音乐）、渐入渐出。
- 锁定 `edit_decisions.render_runtime`，必须与 `proposal_packet` 中用户批准的运行时一致。
- 检查时间线无空隙、无重叠。

产物：`edit_decisions`（含 cuts、overlays、subtitle_config、music_config、render_runtime）。

工具：本阶段通常不直接生成媒体，只做决策；后续由 `video_compose` / `audio_mixer` 执行。

### 4.8 compose — 合成渲染

核心事项：

1. 读取 `edit_decisions.render_runtime`，**禁止静默切换运行时**。
2. 调用对应引擎：
   - **FFmpeg**：简单拼接/裁剪/字幕压制。
   - **Remotion**：React 组件动画、文本卡、图表、弹簧转场、逐字字幕。
   - **HyperFrames**：HTML/CSS/GSAP 高阶动效、产品宣传片、SVG 角色动画。
3. 使用 `audio_mixer` 混合 narration、music、音效。
4. 用 `ffprobe` 验证输出文件合法、时长在目标 ±5% 内。
5. 对 HyperFrames 输出：先 `lint` → `validate` → `render`。

产物：`render_report`（含输出路径、编码配置、验证结果）+ `final_review`。

**常用工具：**

| 工具 | 作用 |
|---|---|
| `video_compose` | 合成总入口，按 `render_runtime` 分派到具体引擎 |
| `hyperframes_compose` | HyperFrames 工作区物化、lint/validate/render |
| `video_stitch` | 多段视频拼接 |
| `audio_mixer` | 音频混音、ducking、音量平衡 |
| `remotion_caption_burn` / `subtitle_gen` | 字幕生成与压制 |

### 4.9 publish — 发布

核心事项：

- 生成/补全 SEO 元数据（标题、描述、标签、缩略图概念）。
- 生成章节标记（chapters）。
- 整理导出包：`final.mp4`、metadata、thumbnail concept、subtitle 文件。
- 如果需要分发到具体平台，按 `lib/media_profiles.py` 中的平台配置（YouTube、TikTok、Instagram 等）再次检查。

产物：`publish_log`。

工具：通常不调用生成工具，以整理和元数据为主。

## 5. 跨阶段支撑机制

### 5.1 状态与续作（Checkpoint）

每个项目 workspace 下有 `pipeline/checkpoint_<stage>.json`，记录：

- 当前阶段、状态（`in_progress` / `completed` / `awaiting_human` / `failed`）
- 当前阶段产物 artifact
- 自检结果
- 成本快照

这样任意阶段失败后，可以从上次 checkpoint 恢复，不必重做。

### 5.2 自检（Reviewer）

每完成一个阶段，Agent 用 `skills/meta/reviewer.md` 进行自检：

- 检查产物是否通过 JSON Schema 验证。
- 检查 pipeline manifest 中 `review_focus` 和 `success_criteria`。
- 最多两轮 revision，之后通过并记录警告。

### 5.3 预算治理（Cost Tracker）

`tools/cost_tracker.py` 贯穿全程：

```
estimate(tool, operation, $) → reserve(entry_id) → reconcile(entry_id, actual_$)
```

- 默认总预算 $2.00（`animated-explainer`）。
- 单步超过阈值（默认 $0.50）需确认。
- 模式：`observe`（仅记录）、`warn`（超支警告）、`cap`（超支拒绝）。

### 5.4 项目 Workspace

所有产物写入 `projects/<project-name>/`：

```
projects/<project-name>/
├── artifacts/          # 各阶段 JSON artifact
├── assets/
│   ├── images/         # 生成图片
│   ├── video/          # 生成视频片段
│   ├── audio/          # 配音 + 最终混音
│   ├── music/          # 背景音乐
│   └── subtitles.srt   # 字幕
└── renders/
    └── final.mp4       # 最终成片
```

命名规范：从视频标题派生 kebab-case，例如 `hidden-math-of-nature`。

## 7. 关键决策点（不可绕过的硬规则）

1. **所有生产必须走管线** —— 不允许直接写临时脚本调用工具。
2. **Preflight 必须做** —— 用户必须知道自己有什么、缺什么。
3. **Proposal 是花钱前的强制审批闸口** —— 未经 `approval.status == approved` 不得进入 assets。
4. **渲染运行时必须在提案时确定并记录** —— 当 Remotion 和 HyperFrames 都可用时，必须同时展示并让用户选。
5. **禁止静默切换 provider / model / render_runtime** —— 任何重大变更必须告知用户并重新审批。
6. **调用生成工具前必须读对应的 Layer 3 Skill** —— 如 `ai-video-gen`、`flux-best-practices`、`elevenlabs` 等。
7. **每个阶段产物必须通过 schema 验证** —— 防止错误向下游传播。
